## 1. Overview and Design Philosophy

The Core Engine is a headless Python daemon that provides the intelligence backbone of the C2 platform. It ingests sensor data from any source, fuses it into a single entity picture, classifies every entity, and emits enriched CoT events to a TAK Server. It has no user interface — it is the data plane that powers the COP.

**Design principles:**
- Every ingest source is a plugin that implements one abstract interface. Adding a new source (radar, SIGINT, HUMINT) requires writing one class and registering it; nothing else changes.
- PostgreSQL + PostGIS is the single source of truth. The in-memory caches (VesselTracker, AircraftTracker) that exist today become write-through caches — they still exist for low-latency classification, but every update is persisted immediately.
- CoT output to TAK Server is the only external output interface. Alerts, position reports, and enriched tracks all leave the system as CoT events.
- Every decision — every classification, every alert trigger, every correlation — is written to the audit log with its full reasoning chain. This is non-negotiable for any system used in military contexts.
- The classifier in `classifier.py` is preserved and promoted. It becomes the Classification Worker, called per-entity after each Track Manager update.

**Runtime model:** asyncio-native daemon with a structured event loop. Ingest receivers run as long-lived async tasks. The Track Manager receives raw observations on an internal asyncio queue. Downstream workers (Classifier, Alert Engine, CoT Output) are chained. The main process handles signals for graceful shutdown.

## 3. PostgreSQL + PostGIS Schema

All tables use `timestamptz` for all timestamps (UTC enforced). PostGIS extension must be installed. UUIDs generated by the database using `gen_random_uuid()`.

### 3.1 `entities` — Resolved entity registry

```sql
CREATE TABLE entities (
    entity_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),

    -- Identity
    display_name    TEXT,
    entity_type     TEXT NOT NULL,           -- 'surface_vessel','fixed_wing','uav','ground_vehicle','person','submarine'
    domain          TEXT NOT NULL,           -- 'air','sea','land','subsurface','space'

    -- Source identifiers (any can be NULL; at least one must be set)
    mmsi            BIGINT UNIQUE,           -- AIS vessel identifier
    icao_hex        CHAR(6) UNIQUE,          -- ADS-B aircraft identifier
    mavlink_sysid   SMALLINT,               -- MAVLink system ID (own assets only)
    cot_uid         TEXT UNIQUE,             -- CoT UID from TAK source
    registration    TEXT,                    -- Aircraft registration e.g. G-ABCD
    imo_number      BIGINT,                  -- IMO vessel number

    -- Static attributes
    callsign        TEXT,
    ship_type       SMALLINT,               -- AIS ship type code 0-99
    aircraft_type   TEXT,                   -- ICAO type designator e.g. B738
    flag_state      CHAR(3),                -- ISO 3166-1 alpha-3

    -- Classification (latest, denormalised for fast queries)
    affiliation     TEXT NOT NULL DEFAULT 'unknown',
    threat_level    SMALLINT NOT NULL DEFAULT 0,
    threat_category TEXT NOT NULL DEFAULT 'none',
    classification_confidence FLOAT,
    classified_at   TIMESTAMPTZ,

    -- Source tracking
    primary_source  TEXT NOT NULL,          -- 'ais','adsb','mavlink','cot','manual'
    source_count    SMALLINT NOT NULL DEFAULT 1,  -- how many sources have reported this entity

    -- State flags
    is_own_force    BOOLEAN NOT NULL DEFAULT false,
    is_active       BOOLEAN NOT NULL DEFAULT true
);

CREATE INDEX idx_entities_mmsi         ON entities(mmsi) WHERE mmsi IS NOT NULL;
CREATE INDEX idx_entities_icao         ON entities(icao_hex) WHERE icao_hex IS NOT NULL;
CREATE INDEX idx_entities_affiliation  ON entities(affiliation);
CREATE INDEX idx_entities_threat       ON entities(threat_level DESC);
CREATE INDEX idx_entities_active       ON entities(is_active) WHERE is_active = true;
```

### 3.2 `tracks` — Time-series position history

Every position update from every source is appended here. This is the raw history log — never updated, only inserted.

```sql
CREATE TABLE tracks (
    track_id        BIGSERIAL PRIMARY KEY,
    entity_id       UUID NOT NULL REFERENCES entities(entity_id) ON DELETE CASCADE,
    observed_at     TIMESTAMPTZ NOT NULL,
    received_at     TIMESTAMPTZ NOT NULL DEFAULT now(),

    -- Position (PostGIS point, SRID 4326 = WGS84)
    position        GEOGRAPHY(POINT, 4326) NOT NULL,
    latitude        FLOAT NOT NULL,
    longitude       FLOAT NOT NULL,
    altitude_m      FLOAT,

    -- Motion
    speed_mps       FLOAT,
    heading_deg     FLOAT,
    vertical_rate_mps FLOAT,

    -- Source
    source          TEXT NOT NULL,          -- 'ais','adsb','mavlink','cot','radar'
    source_quality  FLOAT,                  -- 0.0-1.0 if available
    raw_payload     JSONB                   -- full decoded source message for replay
);

-- Time-series index (most queries filter by entity + time range)
CREATE INDEX idx_tracks_entity_time    ON tracks(entity_id, observed_at DESC);
-- PostGIS spatial index
CREATE INDEX idx_tracks_position       ON tracks USING GIST(position);
-- Recent tracks (last 24h) — partial index for fast live queries
CREATE INDEX idx_tracks_recent         ON tracks(observed_at DESC)
    WHERE observed_at > now() - INTERVAL '24 hours';

-- Partition by month in production (add after initial implementation):
-- PARTITION BY RANGE (observed_at)
```

### 3.3 `entity_current_position` — Materialised latest position per entity

Updated by a trigger on `tracks` insert. Used for all live-picture queries instead of scanning the full tracks table.

```sql
CREATE TABLE entity_current_position (
    entity_id       UUID PRIMARY KEY REFERENCES entities(entity_id) ON DELETE CASCADE,
    updated_at      TIMESTAMPTZ NOT NULL,
    position        GEOGRAPHY(POINT, 4326) NOT NULL,
    latitude        FLOAT NOT NULL,
    longitude       FLOAT NOT NULL,
    altitude_m      FLOAT,
    speed_mps       FLOAT,
    heading_deg     FLOAT,
    source          TEXT NOT NULL,
    track_id        BIGINT REFERENCES tracks(track_id)
);

CREATE INDEX idx_ecp_position ON entity_current_position USING GIST(position);

-- Trigger: upsert entity_current_position on each tracks insert
CREATE OR REPLACE FUNCTION update_entity_current_position()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    INSERT INTO entity_current_position
        (entity_id, updated_at, position, latitude, longitude, altitude_m,
         speed_mps, heading_deg, source, track_id)
    VALUES
        (NEW.entity_id, NEW.observed_at, NEW.position, NEW.latitude, NEW.longitude,
         NEW.altitude_m, NEW.speed_mps, NEW.heading_deg, NEW.source, NEW.track_id)
    ON CONFLICT (entity_id) DO UPDATE SET
        updated_at   = EXCLUDED.updated_at,
        position     = EXCLUDED.position,
        latitude     = EXCLUDED.latitude,
        longitude    = EXCLUDED.longitude,
        altitude_m   = EXCLUDED.altitude_m,
        speed_mps    = EXCLUDED.speed_mps,
        heading_deg  = EXCLUDED.heading_deg,
        source       = EXCLUDED.source,
        track_id     = EXCLUDED.track_id
    WHERE EXCLUDED.updated_at > entity_current_position.updated_at;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_update_ecp
AFTER INSERT ON tracks
FOR EACH ROW EXECUTE FUNCTION update_entity_current_position();
```

### 3.6 `audit_log` — Immutable audit trail

Every significant action or decision is written here. The audit log is append-only — no updates or deletes.

## 4. Daemon Lifecycle

### 4.1 `CoreEngineDaemon` class — `daemon.py`

```
class CoreEngineDaemon:
    Attributes:
        config: EngineConfig
        db_pool: asyncpg.Pool
        track_manager: TrackManager
        classifier: EntityClassifier
        classification_worker: ClassificationWorker
        alert_engine: AlertEngine
        cot_pipeline: CotOutputPipeline
        receivers: list[IngestReceiver]
        _observation_queue: asyncio.Queue[RawObservation]
        _classified_queue: asyncio.Queue[ClassifiedEntity]
        _running: bool

    Methods:
        async start() -> None
            # 1. Load config
            # 2. Connect to PostgreSQL (asyncpg pool, min_size=2, max_size=10)
            # 3. Run schema migrations
            # 4. Load known-entity lists from known_entities table into classifier
            # 5. Load alert rules from alert_rules table into alert engine
            # 6. Create internal asyncio queues
            # 7. Instantiate TrackManager, ClassificationWorker, AlertEngine, CotOutputPipeline
            # 8. Instantiate ingest receivers from config
            # 9. Register signal handlers (SIGTERM, SIGINT → graceful_shutdown)
            # 10. Start all receivers as asyncio tasks
            # 11. Start track_manager.run() as asyncio task
            # 12. Start classification_worker.run() as asyncio task
            # 13. Start alert_engine.run() as asyncio task
            # 14. Start cot_pipeline.run() as asyncio task
            # 15. Start maintenance_loop() as asyncio task
            # 16. Write startup event to audit_log
            # 17. Await asyncio.gather(*all_tasks)

        async graceful_shutdown(signum: int) -> None
            # 1. Log shutdown initiation to audit_log
            # 2. Set _running = False
            # 3. Signal all receivers to stop (call receiver.stop())
            # 4. Wait for observation_queue to drain (timeout: 10s)
            # 5. Cancel all tasks in reverse start order
            # 6. Close db_pool
            # 7. Log shutdown complete

        async maintenance_loop() -> None
            # Runs every 60 seconds:
            # 1. Mark entities as inactive if no position update in N minutes (configurable)
            # 2. Write maintenance heartbeat to audit_log
            # 3. Log active entity count, alert count, queue depths
```

### 4.2 Entry point — `__main__.py`

```
Behaviour:
    1. Parse CLI args: --config <path> (default: /etc/core_engine/config.yaml)
    2. Set up structured logging (JSON format, level from config)
```

## 5. Ingest Pipeline Interface

### 5.1 Abstract base — `ingest/base.py`

```
class IngestReceiver (ABC):
    """Every ingest source implements this interface.

    The receiver is responsible for:
    - Opening its data source (socket, HTTP connection, serial port)
    - Decoding source-specific messages
    - Normalising to RawObservation
    - Putting observations onto the shared queue
    - Handling its own reconnection / error recovery
    - Stopping cleanly when stop() is called
    """

    Attributes:
        name: str                          # Human-readable name, used in logs + audit
        source: TrackSource                # From c2_models.TrackSource enum
        _queue: asyncio.Queue[RawObservation]
        _running: bool
        _config: dict                      # Receiver-specific config block

    Abstract methods:
        async connect() -> None
            # Open connection to data source. Raise ReceiverError on failure.

        async disconnect() -> None
            # Cleanly close connection.

        async _receive_loop() -> None
            # Internal loop: receive messages, decode, normalise, enqueue.
            # Must check self._running and exit when False.
            # Must handle source-specific errors with exponential backoff.

    Concrete methods:
        async run() -> None
            # Called by daemon as asyncio task.
            # 1. Call connect() with retry (3 attempts, exponential backoff)
            # 2. Run _receive_loop()
            # 3. On exception: log, wait backoff_seconds, retry connect + loop
            # 4. On stop() called: call disconnect(), return

        def stop() -> None
            # Set _running = False

        def _emit(observation: RawObservation) -> None
            # Put observation on queue (non-blocking; drop + log if queue full)

    Exceptions:
        ReceiverError(Exception)           # Connection or decode failure
```

### 5.2 `RawObservation` — normalised ingest record — `track/models.py`

```
@dataclass
class RawObservation:
    """A single position/state report from any ingest source.

    All receivers normalise to this structure before publishing to the
    observation queue. The Track Manager never sees source-specific data.
    """
    # Required
    source: TrackSource                    # Origin of this observation
    observed_at: datetime                  # UTC timestamp of the observation itself

    # At least one source identifier must be set
    mmsi: int | None = None               # AIS
    icao_hex: str | None = None           # ADS-B
    mavlink_sysid: int | None = None      # MAVLink
    cot_uid: str | None = None            # CoT

    # Position (all optional — static messages may have no position)
    latitude: float | None = None
    longitude: float | None = None
    altitude_m: float | None = None
    speed_mps: float | None = None
    heading_deg: float | None = None
    vertical_rate_mps: float | None = None

    # Entity attributes (from static/metadata messages)
    display_name: str | None = None
    callsign: str | None = None
    entity_type: EntityType | None = None
    domain: Domain | None = None
    ship_type: int | None = None          # AIS ship type
    aircraft_type: str | None = None      # ICAO type designator
    squawk: str | None = None             # ADS-B transponder code
    nav_status: int | None = None         # AIS navigation status
    on_ground: bool | None = None
    affiliation_hint: Affiliation | None = None  # if source carries affiliation (CoT)

    # Source quality
    source_quality: float = 1.0           # 0.0-1.0 signal/data quality if known
    raw_payload: dict | None = None       # full decoded message for audit/replay
```

### 5.3 Receiver implementations

**`AisReceiver` — `ingest/ais_receiver.py`**
```
Config keys: host (str), port (int, default 10110), filter_mmsi_range (list[tuple] | None)

connect():
    Open asyncio UDP socket bound to host:port (or multicast group).

_receive_loop():
    Loop:
        1. recv UDP datagram
        2. Decode with pyais.decode(raw_nmea)
        3. Extract msg type: type 1/2/3 (position), type 5 (static), type 18/24 (Class B)
        4. Map fields to RawObservation attributes
        5. Set source=TrackSource.AIS, observed_at=datetime.utcnow()
        6. _emit(observation)
```

**`AdsbReceiver` — `ingest/adsb_receiver.py`**
```
Config keys:
    mode: 'http' | 'rtlsdr'
    http_url: str (default 'http://localhost:8080/data/aircraft.json')  # dump1090
    poll_interval_s: float (default 1.0)

connect():
    For 'http' mode: verify HTTP endpoint is reachable (HEAD request).
    For 'rtlsdr': initialise pyModeS decoder (placeholder — not in v1).

_receive_loop():
    For 'http' mode:
        Loop every poll_interval_s:
            1. GET http_url (timeout 3s)
            2. Parse JSON: {"aircraft": [...], "now": ...}
            3. For each aircraft dict:
               a. Map hex→icao_hex, flight→callsign, lat/lon/alt_baro/gs/track/squawk etc.
               b. Build RawObservation
               c. _emit(observation)
```

**`MavlinkReceiver` — `ingest/mavlink_receiver.py`**
```
Config keys:
    connection_string: str (e.g. 'udp:0.0.0.0:14550', 'tcp:192.168.1.1:5760')
    own_force: bool (default true)   # marks entities from this source as is_own_force

connect():
    Connect via pymavlink: mavutil.mavlink_connection(connection_string)

_receive_loop():
    Loop:
        1. Wait for MAVLink message (non-blocking with timeout 0.1s)
        2. On GLOBAL_POSITION_INT:
           Map: lat/1e7→latitude, lon/1e7→longitude, alt/1000→altitude_m,
                vx+vy→speed_mps (derived), hdg/100→heading_deg
        3. On HEARTBEAT:
           Map: type→entity_type (copter=UAV, fixed_wing=FIXED_WING etc.)
        4. Build RawObservation:
           source=TrackSource.MAVLink (add MAVLink to c2_models.TrackSource enum)
           mavlink_sysid=msg.get_srcSystem()
           affiliation_hint=Affiliation.FRIENDLY (own force)
        5. _emit(observation)
```

**`CotReceiver` — `ingest/cot_receiver.py`**
```
Config keys:
    listen_host: str (default '0.0.0.0')
    listen_port: int (default 8087)        # TAK Server CoT ingest port (TCP)
    protocol: 'tcp' | 'udp' | 'multicast'
    multicast_group: str (optional)

connect():
    Open asyncio TCP server or UDP socket per protocol.

_receive_loop():
    For each incoming CoT XML:
        1. Parse XML with xml.etree.ElementTree
        2. Extract from <event>: uid, type, time, stale attributes
        3. Extract from <point>: lat, lon, hae (altitude)
        4. Extract from <detail>/<contact>: callsign
        5. Map CoT type string → entity_type + domain + affiliation_hint
           (e.g. 'a-f-A-M-F-Q' → air, friendly, rotary wing)
        6. Build RawObservation with source=TrackSource.COT
        7. _emit(observation)

    CoT type mapping table (partial):
        'a-f-*' → affiliation=FRIENDLY
        'a-h-*' → affiliation=HOSTILE
        'a-n-*' → affiliation=NEUTRAL
        'a-u-*' → affiliation=UNKNOWN
        '*-A-*' → domain=AIR
        '*-G-*' → domain=LAND
        '*-S-*' → domain=SEA
```

## 6. Track Manager

### 6.1 `TrackManager` — `track/manager.py`

```
class TrackManager:
    """Consumes RawObservations, correlates them to entities, persists, and forwards.

    The TrackManager is the central coordination point. It:
    1. Receives raw observations from the ingest queue
    2. Resolves each observation to an entity (creating one if new)
    3. Persists the track point and updates the entity record
    4. Forwards the updated entity to the classification queue

    It holds an in-memory entity index for fast correlation without DB round-trips
    on every position update.
    """

    Attributes:
        _db: EntityRepository
        _correlator: EntityCorrelator
        _in_queue: asyncio.Queue[RawObservation]       # from ingest receivers
        _out_queue: asyncio.Queue[EntityUpdateEvent]   # to classification worker
        _entity_index: dict[str, EntityRecord]
            # Keyed by all known identifiers:
            # 'mmsi:123456789', 'icao:4CA7B5', 'mavlink:1', 'cot:UID-ABC'
        _running: bool

    Methods:
        async run() -> None
            # Main loop: drain _in_queue, call _process_observation() for each

        async _process_observation(obs: RawObservation) -> None
            # Core logic — see algorithm in section 6.2

        async _resolve_entity(obs: RawObservation) -> EntityRecord
            # Returns existing or newly created EntityRecord

        async _merge_entity(record: EntityRecord, obs: RawObservation) -> EntityRecord
            # Updates entity attributes from observation (non-destructive merge)

        async _persist_track(entity_id: UUID, obs: RawObservation) -> int
            # INSERT into tracks, returns track_id

        async _update_entity_classification_fields(entity_id: UUID, result: ...) -> None
            # UPDATE entities SET affiliation=..., threat_level=..., etc.

        def _index_entity(record: EntityRecord) -> None
            # Add/update all identifier keys in _entity_index

        def _lookup_in_index(obs: RawObservation) -> EntityRecord | None
            # Check _entity_index for any matching identifier from obs
```

### 6.2 Track Manager Algorithm (step by step)

This is the core of the entity resolution problem — the equivalent of Palantir's object resolution.

```
FUNCTION _process_observation(obs: RawObservation):

  STEP 1 — Exact ID lookup (fast path)
    Look up obs in _entity_index using all source identifiers present:
      - if obs.mmsi     → check 'mmsi:{obs.mmsi}'
      - if obs.icao_hex → check 'icao:{obs.icao_hex}'
      - if obs.mavlink_sysid → check 'mavlink:{obs.mavlink_sysid}'
      - if obs.cot_uid  → check 'cot:{obs.cot_uid}'
    IF match found → existing_record = match, go to STEP 4

  STEP 2 — Spatial correlation (new source ID for a known entity)
    Only run if STEP 1 found no match AND obs has a position (lat/lon).
    Call _correlator.find_candidate(obs):
      - Query entity_current_position for entities within 500m of obs position
      - Filter to matching domain (air/sea/land)
      - Filter to entities with position updated in last [stale_timeout] seconds
      - Return closest entity if distance < correlation_threshold (configurable, default 200m)
      - Return None if no candidate or if multiple candidates within threshold (ambiguous)
    IF candidate found:
      - Log to audit_log: "entity_correlated", source, entity_id, distance, reasoning
      - Add new source identifier to entity record (e.g. entity gains icao_hex from ADS-B match)
      - existing_record = candidate, go to STEP 4

  STEP 3 — Create new entity
    No match found. This is genuinely a new entity.
    Call _db.create_entity(obs) → new EntityRecord
    Write audit_log: "entity_created", source, entity_id, all identifiers
    _index_entity(new_record)
    existing_record = new_record

  STEP 4 — Merge attributes
    Call _merge_entity(existing_record, obs):
      Update any non-None fields from obs into existing_record.
      Never overwrite a known value with None.
      If display_name is being updated, log old→new in audit_log.
      If source_count should increase (new source), increment entities.source_count.

  STEP 5 — Persist track point
    If obs has a position (lat IS NOT None):
      Call _persist_track(entity_id, obs)
      (Trigger in DB automatically updates entity_current_position)

  STEP 6 — Update entity record in DB
    Call _db.update_entity(existing_record)
    (Updates display_name, callsign, ship_type, affiliation, updated_at etc.)

  STEP 7 — Emit for classification
    Build EntityUpdateEvent(entity_id, entity_record, observation=obs)
    Put onto _out_queue (non-blocking)
```

### 9.1 `CotOutputPipeline` — `output/cot_pipeline.py`

```
class CotOutputPipeline:
    """Receives Alerts and periodic entity snapshots, emits CoT to TAK Server.

    Two input streams:
    1. Alert queue: high-priority, emitted immediately
    2. Position heartbeat: all active entities, emitted on a configurable interval

    Uses pytak (already a dependency) for transport to TAK Server.
    """

    Attributes:
        _alert_queue: asyncio.Queue[Alert]
        _db: EntityRepository
        _formatter: CotFormatter
        _tak_url: str                      # pytak URL e.g. 'tcp:takserver.lan:8088'
        _heartbeat_interval_s: float       # default 30.0
        _tx_queue: asyncio.Queue           # pytak TX queue
        _running: bool

    Methods:
        async run() -> None
            # 1. Set up pytak CLITool with _tak_url
            # 2. Start two tasks in parallel:
            #    - _alert_sender_loop()
            #    - _heartbeat_loop()
            # 3. Run until stopped

        async _alert_sender_loop() -> None
            # Drain _alert_queue continuously:
            # 1. Pop Alert from queue
            # 2. Fetch entity from DB (for current position + classification)
            # 3. Call _formatter.alert_to_cot(alert, entity)
            # 4. Put CoT XML onto _tx_queue
            # 5. UPDATE alerts SET cot_sent=true in DB
            # 6. Write to audit_log: "alert_cot_sent"

        async _heartbeat_loop() -> None
            # Every _heartbeat_interval_s:
            # 1. Query entity_current_position for all active entities
            #    updated in last (heartbeat_interval * 3) seconds
            # 2. For each entity:
            #    a. Fetch classification from entities table
            #    b. Call _formatter.entity_to_cot(entity, classification, position)
            #    c. Put CoT XML onto _tx_queue
            #    d. Sleep 0.05s between events to avoid flooding
            # (Own-force entities use their own CoT type; tracks use appropriate type)

        async stop() -> None
            # Set _running = False, drain queues, close pytak connection
```

```
class CotFormatter:
    """Serialises enriched entities and alerts to CoT XML.

    This class is the inverse of CotReceiver — it builds CoT XML from
    internal entity records. It extends the existing CoTWriter in
    mpe/writers/cot.py, reusing its XML structure but taking EntityRecord
    rather than MissionItem as input.
    """

        def entity_to_cot(
            entity: EntityRecord,
            classification: Classification,
            position: CurrentPosition,
            stale_seconds: int = 90,
        ) -> str
            # Returns CoT XML string for a position report event.
            # CoT type derived from entity.domain + entity.entity_type + classification.affiliation:
            #   friendly air uav     → 'a-f-A-M-F-Q'
            #   hostile sea vessel   → 'a-h-S-X-X'
            #   neutral air fixed    → 'a-n-A-C-F'
            #   unknown              → 'a-u-G-E-X' etc.
            # CoT <detail> includes:
            #   <contact callsign="..." />
            #   <ce_metadata threat_level="..." affiliation="..." confidence="..."
            #                 source_count="..." anomalies="..." />
            # (ce_metadata is a custom namespace — understood by custom ATAK plugins)

        def alert_to_cot(
            alert: Alert,
            entity: EntityRecord | None,
        ) -> str
            # Returns CoT XML string for an alert event (CoT type 'b-...' for emergency/alarm).
            # CoT type mapping:
            #   AlertType.EMERGENCY    → 'b-a-o-tbl'  (emergency)
            #   AlertType.THREAT       → 'b-t-...'    (tactical)
            #   AlertType.ANOMALY      → 'b-i-...'    (information/intel)
            #   AlertType.INFORMATION  → 'b-i-...'
            # Position from entity.current_position if available; else 0,0,0 (alerter position).
            # Alert description in CoT <remarks> element.

        def _affiliation_to_cot_prefix(affiliation: Affiliation) -> str
            # 'friendly' → 'a-f', 'hostile' → 'a-h', 'neutral' → 'a-n',
            # 'suspect' → 'a-s' (non-standard, but used in practice), 'unknown' → 'a-u'

        def _domain_entity_to_cot_suffix(domain: Domain, entity_type: EntityType) -> str
            # Lookup table → CoT type suffix fragment
            # air + uav → 'A-M-F-Q'
            # air + fixed_wing → 'A-C-F'
            # air + rotary_wing → 'A-M-H'
            # sea + surface_vessel → 'S-X-M'
            # land + ground_vehicle → 'G-E-V-C'
            # etc.
```

## 13. Internal Queue Architecture

The five queues that connect the pipeline stages:

```
receivers[*] → _observation_queue (RawObservation)
                        ↓
            TrackManager._process_observation()
                        ↓
              _entity_update_queue (EntityUpdateEvent)
                        ↓
          ClassificationWorker._classify()
                        ↓
             _classified_queue (ClassifiedEntity)
                        ↓
              AlertEngine._evaluate()
                        ↓
               _alert_queue (Alert)
                        ↓
           CotOutputPipeline._alert_sender_loop()
                        ↓
                _tak_tx_queue (bytes)
                        ↓
                TAK Server (pytak)


Heartbeat path (parallel to alert path):
CotOutputPipeline._heartbeat_loop() reads from DB directly
                        ↓
                _tak_tx_queue (bytes)
                        ↓
                TAK Server (pytak)
```

All queues use `asyncio.Queue` with `maxsize` set in config (default 1000). If a queue reaches capacity, the producer logs a warning and drops the item — backpressure is logged but not propagated, so a slow downstream stage does not block ingest.

## 15. Migration from Existing Modules

The existing modules in `mpe/` are preserved and promoted, not replaced:

| Existing module | Role in Core Engine |
|---|---|
| `c2_models.py` | `EntityType`, `Domain`, `Affiliation`, `TrackSource`, `Task`, `TaskPlan` enums and models are imported directly by core_engine. Add `TrackSource.MAVLINK` to the enum. |
| `classifier.py` | `EntityClassifier` and `Classification` are imported and wrapped by `ClassificationWorker`. No changes needed to `classifier.py`. |
| `vessel_tracker.py` | `VesselTracker` is kept as a fast in-memory read cache for MAVLink/mission queries. `EntityRepository.get_entities_near()` replaces it for the Core Engine's spatial queries. Both can coexist. |
| `aircraft_tracker.py` | Same pattern as vessel_tracker. |
| `cot_sender.py` | `CoTWriter` from `mpe/writers/cot.py` is reused by `CotFormatter`. `CoTStreamer` is not used (replaced by `CotOutputPipeline`). |

The Core Engine lives at `core_engine/` as a sibling package to `mpe/`, sharing the same repo. It imports from `mpe` where useful.

## 16. Future Extension Points

The following are **explicitly not in v1** but the architecture accommodates them without breaking changes:

1. **Kalman filter** — plug in at `EntityCorrelator`; replace the simple nearest-neighbour with a Kalman-predicted position before running the spatial query. `EntityRecord` gains a `kalman_state` field.

2. **Pattern of life** — add a `PatternOfLifeAnalyser` worker that reads from the `tracks` table for a sliding 7-day window per entity. It writes anomalies of type `pattern_deviation` into the `classifications.anomalies` JSONB field. The `AlertEngine` needs no changes — it already handles `anomaly_types_any`.

3. **ML classifier** — the `ClassificationWorker._classify()` method calls `EntityClassifier` methods. Replace those methods with a model inference call that returns the same `Classification` dataclass. The pipeline is unchanged.

4. **SIGINT / HUMINT / OSINT** — add a `HumintReceiver` and `SigintReceiver` that implement `IngestReceiver`. They produce `RawObservation` with `source=TrackSource.SIGINT`. The Track Manager sees no difference.

5. **REST API** — add a FastAPI app that reads from the same PostgreSQL database. It is a separate process from the daemon. The database is the integration point.

6. **Webhook output** — `AlertEngine` already has `send_webhook` and `webhook_url` fields on `AlertRule`. A `WebhookSender` worker is added to the pipeline alongside `CotOutputPipeline`, consuming from the same `_alert_queue`.

7. **Multi-classification** — as more allied data sources come online (radar, EO/IR), `source_count` in `entities` grows. Higher `source_count` → higher classification confidence. This is already modelled; the `EntityClassifier` just needs to be extended to weight confidence by source count.
